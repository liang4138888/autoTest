# tc-brainstorm

## Purpose

`tc-brainstorm` 是设计阶段测试流程入口，负责把 PRD、TRD、接口文档或需求描述转化为可继续生成计划和用例的测试分析结果。

## When to use

当用户提供以下任一材料，并希望开始测试设计时使用：

- PRD 文件或需求描述
- TRD，可选
- 接口文档，可选
- 历史 Bug 信息，可选
- 业务背景说明，可选

## Inputs

```text
PRD 文件或需求描述
TRD 文件，可选
接口文档，可选
历史 Bug 信息，可选
业务背景说明，可选
```

## Outputs

主输出：

```text
test-design/strategy/test_brainstorm.md
```

标准 / 完整模式下，输出中必须包含：

```text
复杂度判断
推荐流程模式
风险清单
待确认问题
```

## Required sections

`test_brainstorm.md` 应包含：

1. 需求背景。
2. 测试目标。
3. 业务流程拆解。
4. 测试范围。
5. 不测范围。
6. 暂不可测范围。
7. 核心业务规则。
8. 风险点识别。
9. 测试类型建议。
10. 待确认问题。
11. 推荐测试模式：轻量 / 标准 / 完整。

## Execution rules

1. 默认按轻量模式判断。
2. 满足多模块、多接口、数据准备、排期评估、明显风险等条件时，推荐标准模式。
3. 满足 P0/P1 核心链路、自动化沉淀、多系统联动、高风险域、持续回归、执行报告等条件时，推荐完整模式。
4. 缺失关键业务规则时，先输出待确认问题，不要编造业务规则。
5. 如果输入材料不足以生成用例，应明确阻塞项。
6. 输出必须标明来源材料。

## Review gate

```text
Gate：策略 / 范围确认
```

确认内容：

1. 测试范围是否合理。
2. 不测范围是否明确。
3. 风险识别是否完整。
4. 待确认问题是否需要先补齐。
5. 推荐模式是否合理。

## Next step

- 标准模式：进入 `tc-writing-plan`。
- 完整模式：进入 `tc-writing-plan`。
